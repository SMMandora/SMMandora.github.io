Context

Real-time mobile game with a global player base on Google Play and the App Store. Hard requirements: 24×7 availability, leaderboard consistency under contention, transactional integrity for in-game purchases, and low-latency gameplay APIs.

Approach

Multi-region distributed backend on AWS. Microservices decomposed by domain — gameplay, auth, leaderboard, financial transactions — each with its own test suite (unit, integration, contract, performance). CI/CD via Docker, Jenkins, GitHub Actions, and Kubernetes with automated build verification and regression gates. Telemetry-driven observability with SLI/SLO dashboards in Grafana and proactive alerting.

Outcome

SLO-backed availability with automated failure detection. Proactive bottleneck detection through load and stress testing under production-like conditions. Release confidence from quality gates blocking regressions before they ship.

  • Context

    Code review is the bottleneck on small teams, and generic AI reviewers miss repo-specific conventions.

    Approach

    GitHub webhook into an HMAC-validated FastAPI receiver. A LangGraph state machine fans out to four parallel check nodes, grounded by pgvector retrieval over past PR comments and style guides. Findings dedupe into one consolidated GitHub review.

  • Context

    Real-time fan-out at scale is the canonical distributed-systems problem: connections are stateful, everything else shouldn't be.

    Approach

    Stateless Go gateways behind an L4 balancer with Redis pub/sub for cross-replica fan-out. A persistence worker batches writes to Postgres. Token-bucket rate limiting, presence via Redis sorted sets, graceful drain on shutdown.

    Target

    10k concurrent connections per gateway replica with p99 fan-out latency under 50ms, verified by k6 load tests and traced end-to-end with OpenTelemetry.

  • Context

    Approach

    Three services with injectable failure modes feed Prometheus, Loki, and Tempo. Grafonnet-rendered dashboards track SLIs, error budgets, and 1h/6h/24h burn rates. Alerts are unit-tested with promtool, and each links to its runbook.

    Target

    Chaos scenarios that auto-verify the right alert fires — kill pods, inject latency, fill disks — with the full stack up locally in under two minutes.

devops practice

  1. 01

    Quality gates, not announcements

    Automated unit, integration, contract, and performance suites block regressions before they reach prod — they don't just notify after.

  2. 02

    Observability designed in, not bolted on

    SLI/SLO from day one. Grafana dashboards and proactive alerting paired with every service, not added after the first incident.

  3. 03

    Reproducible by default

    Docker and K8s declarative configs. CI is the only path to prod — local-only fixes don't ship.

  4. 04

    Test under load before users do

    Load and stress testing against production-like conditions catches bottlenecks before real traffic surfaces them.
